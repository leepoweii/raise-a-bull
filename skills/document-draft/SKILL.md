# Document Draft Skill

幫助使用者撰寫各類辦公室文件，從公告到簽呈，產出格式正確、語氣適當的草稿。

---

## 支援文件類型

### 1. 公告（Announcement）

**用途：** 對內或對外發布消息、政策、異動等。

**標準結構：**
```
主旨：（一句話說明公告重點）
說明：
一、（背景或原因）
二、（具體內容）
三、（執行方式或注意事項）
聯絡人：（姓名、電話、Email）
```

**必填欄位：** 主旨、說明內容、發布單位、日期
**語氣：** 正式公文語氣

---

### 2. 邀請函（Invitation）

**用途：** 邀請出席活動、會議、典禮等。

**標準結構：**
```
受文者：（稱謂 + 姓名）

（開頭敬語）

敬邀您出席「（活動名稱）」，相關資訊如下：

一、時間：YYYY 年 MM 月 DD 日（星期X）HH:MM
二、地點：（地址或線上連結）
三、活動內容：（簡述）
四、報名方式：（如適用）

敬請撥冗出席，不勝感荷。

（署名）
（單位名稱）
（日期）
```

**必填欄位：** 受文者、活動名稱、時間、地點、署名單位
**語氣：** 正式但溫暖，可依對象調整親和度

---

### 3. 通知 / 備忘錄（Notice / Memo）

**用途：** 內部通知事項、提醒、行政聯繫。

**標準結構：**
```
受文者：（對象或單位）
發文者：（單位或個人）
日期：YYYY 年 MM 月 DD 日
主旨：（簡述通知事項）

說明：
一、（通知內容）
二、（需配合事項）
三、（截止日期或注意事項）

如有疑問，請洽（聯絡人）。
```

**必填欄位：** 受文者、發文者、主旨、說明內容
**語氣：** 正式簡潔

---

### 4. 會議紀錄（Meeting Minutes）

**用途：** 記錄會議討論內容、決議事項、待辦追蹤。

**標準結構：**
```
會議名稱：
會議時間：YYYY 年 MM 月 DD 日（星期X）HH:MM - HH:MM
會議地點：
主席：
出席人員：
記錄：

一、報告事項：
  （一）...
  （二）...

二、討論事項：
  案由一：（議題）
  說明：（背景）
  決議：（結論）

  案由二：...

三、臨時動議：

四、散會時間：HH:MM

待辦事項追蹤：
| 項次 | 事項 | 負責人 | 完成期限 |
|------|------|--------|----------|
| 1    |      |        |          |
```

**必填欄位：** 會議名稱、時間、出席人員、討論事項、決議
**語氣：** 客觀紀實，不加主觀評論

---

### 5. 報告（Report）

**用途：** 工作報告、成果報告、調查報告、進度報告。

**標準結構：**
```
報告名稱：
報告人：
報告日期：YYYY 年 MM 月 DD 日
報告期間：（如適用）

壹、前言 / 背景
（說明報告目的與背景）

貳、執行內容
一、...
二、...

參、成果與數據
（量化成果、關鍵指標）

肆、問題與建議
一、遭遇問題：
二、改善建議：

伍、結語 / 後續計畫
```

**必填欄位：** 報告名稱、報告人、日期、執行內容
**語氣：** 正式、條理分明，善用數據佐證

---

### 6. 活動企劃（Event Proposal）

**用途：** 提案活動規劃，爭取核准或資源。

**標準結構：**
```
活動名稱：
提案單位：
提案日期：YYYY 年 MM 月 DD 日

壹、活動目的
（為什麼要辦這個活動）

貳、活動概要
一、時間：
二、地點：
三、對象：
四、預計人數：

參、活動內容與流程
| 時間 | 內容 | 負責人 |
|------|------|--------|
|      |      |        |

肆、所需資源
一、人力：
二、場地：
三、設備 / 物資：

伍、預算表
| 項目 | 數量 | 單價 | 小計 |
|------|------|------|------|
|      |      |      |      |
| 合計 |      |      |      |

陸、預期效益

柒、備註
```

**必填欄位：** 活動名稱、目的、時間地點、內容流程、預算
**語氣：** 專業但有說服力

---

### 7. 簽呈 / 簽陳（Official Request）

**用途：** 向上級請示、核准事項，正式行政簽核文件。

**標準結構：**
```
簽呈

受文者：（主管職稱 + 姓名）
簽呈人：（職稱 + 姓名）
日期：中華民國 OOO 年 OO 月 OO 日

主旨：（請示或報告事項，一句話）

說明：
一、（背景與原因）
二、（具體事項）
三、（預期效益或影響）

擬辦：
（具體建議做法，例如「擬請核准辦理」）

附件：（如有）

簽呈人：____________
單位主管：____________
核准：____________
```

**必填欄位：** 受文者、簽呈人、主旨、說明、擬辦
**語氣：** 嚴謹正式公文體

## 執行流程

1. **確認文件類型**
   - 詢問使用者要撰寫哪一種文件
   - 如果使用者描述模糊，根據內容判斷最適合的類型並確認

2. **蒐集關鍵資訊**
   - 根據文件類型的必填欄位，逐一詢問：
     - **Who** — 發文者、受文者、相關人員
     - **What** — 主旨、具體內容
     - **When** — 日期、時間、期限
     - **Where** — 地點（如適用）
     - **Why** — 目的、背景原因
   - 不要一次問太多，分批詢問

3. **撰寫草稿**
   - 使用正體中文撰寫
   - 套用對應文件類型的標準結構
   - 在對話中直接呈現草稿

4. **呈現並請求回饋**
   - 完成後請使用者檢閱
   - 標示可能需要補充或確認的部分（用 `[待確認]` 標記）

5. **修訂**
   - 根據使用者回饋修改
   - 可多輪修訂直到滿意

6. **儲存（如需要）**
   - 如果使用者要求存檔，寫入工作區指定位置

## 語氣與風格規則

### 預設語氣：正式公文體
- 使用「貴」「敬」「惠」「茲」等敬語
- 段落分明，善用「一、二、三」條列
- 避免口語化用詞

### 輕鬆但專業模式（使用者指定時）
- 減少公文用語，改用平易近人的表達
- 保持結構完整，但語氣親和
- 適合內部非正式通知、社群公告

### 組織資訊
- 如果 `params.json` 存在，讀取組織名稱、地點等資訊自動帶入
- 如果是金門相關文件，參照 identity files 使用在地用語

---

## 硬性規則

1. **一律使用正體中文**撰寫所有文件內容
2. **不主動填入機密資訊**（金額、個資、帳號等），除非使用者明確提供
3. **不確定就問**——寧可多問一句，也不要假設細節
4. **日期格式**：預設使用西元（YYYY 年 MM 月 DD 日），簽呈類使用民國紀年
5. **不擅自增加內容**——依照使用者提供的資訊撰寫，不編造事實
6. **標記不確定處**——用 `[待確認]` 或 `[請補充]` 明確標示需要使用者填入的地方
